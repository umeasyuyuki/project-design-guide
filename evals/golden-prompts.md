# Golden Prompts

## Success cases

- 「ECサイトを作りたいんだけど、何から始めればいい？」
  → Phase 0〜1のヒアリングが開始され、5W1Hに基づく質問が提示されること。

- 「社内の勤怠管理システムをWebアプリで作りたい。5人くらいのチームで使う予定」
  → ヒアリング→要件定義→仕様書→DB設計の順に対話が進むこと。
  → 各ドキュメントに教育コメント（なぜこうしたか？）が含まれること。

- 「LINEボットで予約管理システムを作りたい」
  → システム構成図がMermaidで生成されること。
  → LINE Messaging APIとの連携が仕様書に含まれること。
  → DB設計でRDB選定理由が解説されること。

- 「友達と一緒に使える割り勘アプリを作りたい」
  → 個人開発規模として適切な技術スタックが提案されること。
  → 「もっと学ぶなら」セクションが各フェーズに含まれること。

- 「既存の要件定義書をレビューしてほしい」
  → 要件定義書のレビュー＋改善提案として対応すること（全フェーズを実行しない）。

### 分析モード（Mode B）

- 「このプロジェクトのドキュメントを作ってほしい」（既存リポジトリ内で実行）
  → 分析モードが自動選択されること。
  → Phase B0でディレクトリ構成・技術スタックが自動検出されること。

- 「今のコードの問題点を洗い出して、改善案付きのドキュメントを作りたい」
  → Phase B1でコードレビューが実施され、Critical/Warning/Infoに分類されること。
  → 各指摘に教育コメント（なぜ問題か？もっと学ぶなら）が含まれること。

- 「このリポジトリの設計書がないから作りたい」
  → 既存コードからの逆引きで要件定義書が作成されること。
  → 機能要件に「実装状況」カラムが含まれること。

- 「DBの設計を見直したい」
  → 既存のモデル定義/マイグレーションからER図が自動生成されること。
  → 型選定・インデックス・制約に関するレビュー結果が含まれること。

- 「設計書と改善提案の両方がほしい」
  → 4つのファイルが出力されること（requirements, specification, database_design, code_review）。
  → 仕様書に「現状」と「改善後（提案）」の構成図が並記されること。

### 実装モード（Mode C）

- 「設計書ができたので実装して」（docs/ に3つの設計書がある状態で実行）
  → 実装モードが自動選択されること。
  → Phase C0 で設計書を読み込み、タスク分解が行われること。
  → 実装計画書（implementation_plan）が docs/ に出力されること。

- Mode A 完了後に「実装も進めたい」
  → Mode A の Phase 5 完了後に「実装に進みますか？」と提示されること。
  → 承認後、Phase C0 から開始されること。
  → 直前に作成した設計書が自動的に読み込まれること。

- Mode B 完了後に「改善を含めて実装したい」
  → Mode B の Phase B5 完了後に「実装に進みますか？」と提示されること。
  → 承認後、Phase C0 から開始されること。
  → コードレビューの改善提案を反映した実装計画が作成されること。

- 「ユーザー認証APIから実装を始めて」（Phase C2 中のタスク指定）
  → 指定されたタスクの仕様を仕様書から引用して表示すること。
  → TDD サイクル（RED→GREEN→REFACTOR）で実装が進むこと。
  → テストが先に書かれ、失敗を確認してから実装に入ること。

- 「全機能実装したので検証して」（Phase C3）
  → 6段階の品質ゲート（Build, Type, Lint, Test, Security, 設計整合性）が実行されること。
  → 検証レポート（verification_report）が docs/ に出力されること。
  → 総合判定（READY / NOT READY）が明示されること。

- 「通知機能を追加したい」（Phase C4: 機能追加）
  → Step 1 で追加要件の軽量ヒアリングが行われること。
  → Step 2 で設計書（要件定義書・仕様書・DB設計書）が先に更新されること。
  → 設計書のバージョンがインクリメントされること（v1→v2）。
  → Step 3 で実装計画書に新タスクが追加されること。
  → Step 4 で TDD サイクルにより実装されること。
  → Step 5 でリグレッション込みの統合検証が行われること。

- 「まだ途中だけど一回検証して」（Phase C3 の途中実行）
  → 実装済みタスクの範囲で検証が実行されること。
  → 未実装機能は「設計整合性」で未実装として報告されること。

## Failure probes

- 「Reactのコンポーネントを作って」
  → 設計ドキュメント作成ではなく実装タスクのため、このスキルの対象外と判断すること。
  → ただし、docs/ に設計書があり実装モードの文脈であれば対応すること。

- 「100人規模のエンタープライズシステムの全設計を1回で出力して」
  → 規模が大きすぎる場合はフェーズ分割を提案し、一度に全出力しないこと。

- 機密情報（APIキー、パスワード等）がユーザーの入力に含まれている場合
  → ドキュメントにハードコードせず、Secret Manager等の利用を推奨すること。

- docs/ に設計書がない状態で「実装して」と言われた場合
  → 実装モードの前提条件（設計書の存在）が満たされていないことを説明すること。
  → Mode A（新規）または Mode B（分析）の実行を案内すること。

- 「テストは書かなくていいから実装だけして」
  → TDD は実装モードの必須プロセスであることを説明すること。
  → テストを書くことの利点を教育コメント形式で伝えること。
  → ただしユーザーが強く要望する場合は、テストなしのリスクを明示した上で対応すること。
