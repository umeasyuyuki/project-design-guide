# project-design-guide

## Purpose

プロのエンジニアの視点で要件定義書・仕様書・DB設計書を **ユーザーとの対話を通じて段階的に** 作成し、
さらにその設計書を元に **TDDベースの実装** まで一貫して遂行する。

**3つのモード** を提供する:
- **新規モード**: 「こういうシステムを作りたい」というふわっとした要件から設計を起こす
- **分析モード**: 既存プロジェクトのコードを読み解き、疑問点・改善点を洗い出しながらドキュメント化する
- **実装モード**: 作成済みの設計書（要件定義書・仕様書・DB設計書）を元に、タスク分解→TDD実装→検証の一貫フローで開発する

各設計判断には「なぜこの選択をしたか」の理由と、
「ここをもっと学びたいならこの分野を勉強すると良い」というアドバイスを併記する。

## Trigger

- 「要件定義を作りたい」「仕様書を書きたい」「DB設計をしたい」
- 「こういうシステムを作りたいんだけど」「アプリを作りたい」
- 「設計から始めたい」「何から始めればいいかわからない」
- 「既存のプロジェクトのドキュメントを作りたい」「コードを分析してほしい」
- 「このプロジェクトの設計を見直したい」「改善点を洗い出したい」
- 「設計書を元に実装して」「開発を始めたい」「TDDで実装したい」
- 「機能を追加したい」「新しい機能を実装して」（設計書が存在する場合）
- `/project-design-guide` を直接呼び出した場合

## Mode Selection

スキル起動時に、ユーザーの発言内容からモードを判定する:

- **既存のリポジトリやコードベースがある** → 分析モード
- **「作りたい」「始めたい」等の新規開発意図** → 新規モード
- **`docs/` に設計書が存在 + 「実装して」「開発して」等の実装意図** → 実装モード
- **`docs/` に設計書が存在 + 「機能追加したい」等の追加開発意図** → 実装モード（Phase C4 から開始）
- **Mode A/B の最終フェーズ完了後** → 「このまま実装に進みますか？」と提示し、承認されれば実装モードへ遷移
- **判断がつかない場合** → ユーザーに以下を提示して選択してもらう:
  ```
  どのモードで進めますか？

  A) 新規モード — ふわっとした要件から設計書をゼロから作る
  B) 分析モード — 既存プロジェクトのコードを読んで設計書を作る（改善提案付き）
  C) 実装モード — 作成済みの設計書を元に TDD で開発する
  ```

# 新規モード (Mode A)

本モードは **5つのフェーズ** を順番に進める。各フェーズの最後でユーザーの承認を得てから次に進む。

---

### Phase 0: プロジェクト情報の確認

スキル起動時に以下を確認する:

1. **出力先の特定**: スキルを呼び出したリポジトリの `docs/` フォルダを出力先とする。
   - リポジトリのルートを検出し、`<repo-root>/docs/` に出力する。
   - `docs/` フォルダが存在しない場合は作成する。
2. **ファイル命名**: `<slug>_requirements_v1.md`, `<slug>_specification_v1.md`, `<slug>_database_design_v1.md`
   - `<slug>` はプロジェクト名から自動生成（英数字+アンダースコア）。

---

### Phase 1: ヒアリング（要件の深掘り）

**目的**: ユーザーのふわっとした要件を、開発可能な粒度まで具体化する。

**進め方**:
1. ユーザーの初期説明を受け取る。
2. 以下の観点で不足情報を **質問形式** で洗い出す（一度に3〜5個程度）:
   - **Who**: 誰が使うか？（エンドユーザー、管理者、規模感）
   - **What**: 何ができるか？（コア機能、優先度）
   - **Why**: なぜ作るか？（課題、解決したいこと）
   - **Where**: どこで使うか？（Web、モバイル、LINE等）
   - **When**: いつまでに？（スケジュール感）
   - **How**: 予算感、技術的制約、チーム構成
3. 回答を受けて追加質問。2〜3往復で十分な情報を集める。
4. **不明点は仮置きでOK** — 「ここは後で決めましょう」と明示する。

**出力**: ヒアリングで得た情報のサマリーを提示し、ユーザーに確認を求める。

---

### Phase 2: 要件定義書の作成

**目的**: ヒアリング結果を基に、要件定義書を作成する。

**ドキュメント構成** (`references/requirements-template.md` に準拠):

```
1. 概要（目的・対象・背景）
2. スコープ（対象範囲 / 除外範囲）
3. 機能要件（優先度付き）
4. 非機能要件（可用性・性能・セキュリティ等）
5. データ/ナレッジ設計（概要レベル）
6. 外部インターフェース
7. 環境/構成（想定）
8. スケジュール（目安）
9. 受入基準（UAT）目標
10. 運用/保守
11. 前提/制約
12. 見積/費用（概算がある場合）
13. 変更管理
14. リスク/対応
15. 添付/参照
```

**教育セクションのルール**:
- 各セクションの末尾に `> **なぜこうしたか？**` ブロックを配置する。
- 技術選定の理由を「〇〇と△△を比較して、□□の理由で〇〇を選んだ」の形式で記述する。
- `> **もっと学ぶなら:** ` で関連する学習トピックを提示する。

**進め方**:
1. ドキュメントのドラフトを生成してユーザーに提示する。
2. ユーザーのフィードバックを受けて修正する。
3. 承認を得たらファイルに書き出す。

---

### Phase 3: 技術選定とアーキテクチャ設計

**目的**: 要件を実現するための技術スタックとシステム構成を決定する。

**進め方**:
1. 要件定義書の内容から、以下を提案する:
   - **言語/フレームワーク**: 比較表付きで理由を説明
   - **データベース**: RDB vs NoSQL の判断理由を明示
   - **クラウド/インフラ**: 選択肢と推奨理由
   - **外部サービス/API**: 必要な連携先
2. システム構成図を **Mermaid記法** で作成する。
3. ユーザーに選択肢を提示し、合意を得る。

**教育ポイントの例**:
- 「なぜRDB（PostgreSQL）を選んだか？」→ データの関係性が明確、JOIN操作が必要、ACID特性が重要な場合に強い
- 「なぜCloud Runを選んだか？」→ コンテナベースで柔軟、スケールが自動、使った分だけ課金
- 各選定に対して代替技術と比較した上での判断理由を記述する

---

### Phase 4: 仕様書の作成

**目的**: 技術選定を踏まえた詳細仕様書を作成する。

**ドキュメント構成** (`references/specification-template.md` に準拠):

```
1. はじめに（目的・概要・対象読者・関連ドキュメント）
2. システム概要
   2.1. システム目的
   2.2. システムスコープ（スコープ内/外）
   2.3. システム構成図（Mermaid）
   2.4. シーケンス図（Mermaid） ※主要フローごと
3. 機能仕様（機能ごとに詳細記述）
   - 機能概要
   - 入出力仕様
   - 処理仕様（ステップバイステップ）
   - エラーケース
4. 非機能仕様
   4.1. 可用性
   4.2. 性能
   4.3. スケーラビリティ
   4.4. セキュリティ
   4.5. 監査/記録
   4.6. ネットワーク構成（必要な場合）
5. データ仕様
6. 外部インターフェース
7. テスト/受入基準
8. 運用/保守
9. 潜在的なリスクと対策（シナリオ別）
10. 命名規則（リソース命名、ブランチ戦略等）
11. 障害検知とエラーハンドリング
   - エラーシナリオ一覧（表形式）
   - エラー処理フロー（Mermaid）
   - ロギング/監視方針
```

**教育セクションのルール**:
- 技術的な設計判断のたびに `> **設計判断メモ:**` ブロックで理由を解説する。
- シーケンス図やフローチャートには、処理の意図を日本語コメントで補足する。
- `> **もっと学ぶなら:** ` で関連する学習トピック・キーワードを提示する。

**進め方**:
1. セクションごとにドラフトを提示（長大になるため、2〜3セクションずつ）。
2. ユーザーのフィードバックを受けて修正。
3. 全セクション承認後、ファイルに書き出す。

---

### Phase 5: DB設計書の作成

**目的**: アプリケーションのデータモデルを設計し、文書化する。

**ドキュメント構成** (`references/database-design-template.md` に準拠):

```
1. 永続化層の技術選定
   - RDB vs NoSQL の判断理由
   - 開発フロー（ローカル→本番の移行計画）
   - ORM選定と理由
2. ER図（Mermaid erDiagram）
3. テーブル定義
   - テーブルごとに: カラム名 / 型 / 説明 / 制約 を表形式で記述
   - 各テーブルの目的を1行で説明
4. インデックス設計（必要な場合）
5. マイグレーション方針
```

**教育セクションのルール**:
- テーブル設計の各判断に `> **設計判断メモ:**` を付与する。
  - 例: 「なぜ `role` カラムを VARCHAR にしたか？→ ENUMより拡張が容易」
  - 例: 「なぜ `created_at` と `updated_at` を全テーブルに付けるか？→ デバッグ・監査に必須」
- 正規化の判断理由を明示する（「第3正規形まで正規化した」等）。
- `> **もっと学ぶなら:** ` で関連する学習トピックを提示する。

**進め方**:
1. ER図のドラフトを提示。
2. テーブル定義の詳細を提示。
3. ユーザーのフィードバックを受けて修正。
4. 承認後、ファイルに書き出す。

**Mode C への遷移**:
Phase 5 完了後、以下を提示する:
```
設計書（要件定義書・仕様書・DB設計書）が完成しました。
このまま実装に進みますか？

→ はい: 実装モード（Mode C）の Phase C0 に進みます
→ いいえ: ここで終了します（後から Mode C を呼び出せます）
```

# 分析モード (Mode B)

既存プロジェクトのコードを読み解き、プロのエンジニアの視点で **疑問点・改善点を洗い出しながら**
要件定義書・仕様書・DB設計書を生成する。

本モードは **6つのフェーズ** を順番に進める。各フェーズの最後でユーザーの承認を得てから次に進む。

---

### Phase B0: プロジェクトスキャン

**目的**: 既存プロジェクトの全体像を把握する。

**進め方**:
1. **リポジトリルートの特定**: ユーザーが指定したパス、またはカレントディレクトリのリポジトリルートを検出する。
2. **コードベース調査**: 以下を自動で読み取る:
   - ディレクトリ構成（`tree` 相当）
   - `package.json`, `requirements.txt`, `pyproject.toml`, `go.mod` 等の依存定義
   - `Dockerfile`, `docker-compose.yml` 等のインフラ定義
   - `.env.example`, 環境変数の利用箇所
   - DB関連ファイル（マイグレーション、モデル定義、スキーマ）
   - API定義（ルーティング、エンドポイント）
   - 既存ドキュメント（`docs/`, `README.md`）
3. **技術スタックの自動判定**: 言語、フレームワーク、DB、クラウドサービスを一覧化する。
4. **出力先の確認**: `<repo-root>/docs/` に出力。既存ドキュメントがある場合はバージョン番号をインクリメント。

**出力**: スキャン結果のサマリーをユーザーに提示し、認識が正しいか確認する。

---

### Phase B1: コードレビュー＆疑問点の洗い出し

**目的**: プロのエンジニアの目線でコードを読み、疑問点・潜在的な問題・改善機会を洗い出す。

**レビュー観点**（各観点をチェックリスト形式で報告する）:

#### アーキテクチャ
- [ ] 責務の分離は適切か？（コントローラーにビジネスロジックが混在していないか）
- [ ] レイヤー構造は明確か？（ルーティング / ビジネスロジック / データアクセス）
- [ ] コンポーネント間の依存関係は適切か？（循環依存がないか）
- [ ] スケーラビリティを阻害する設計はないか？

#### セキュリティ
- [ ] 認証情報（APIキー、パスワード等）がハードコードされていないか？
- [ ] SQLインジェクション、XSS等の脆弱性はないか？
- [ ] 入力バリデーションは適切か？
- [ ] 認証・認可の仕組みは整っているか？

#### データベース
- [ ] テーブル設計は正規化されているか？（過剰・不足の両面）
- [ ] インデックスは適切か？（N+1問題の可能性は？）
- [ ] マイグレーション管理はされているか？
- [ ] データの整合性制約（外部キー、NOT NULL等）は適切か？

#### エラーハンドリング
- [ ] エラーが握りつぶされていないか？（空の `catch` ブロック等）
- [ ] ユーザーに適切なエラーメッセージが返されるか？
- [ ] リトライ戦略はあるか？（外部API呼び出し時）
- [ ] ロギングは十分か？

#### コード品質
- [ ] 命名規則は統一されているか？
- [ ] 重複コード（DRY違反）はないか？
- [ ] テストは書かれているか？カバレッジは？
- [ ] 未使用のコード・依存パッケージはないか？

#### インフラ・運用
- [ ] 環境変数の管理方法は適切か？
- [ ] CI/CDパイプラインは整備されているか？
- [ ] ログの出力方針は定まっているか？
- [ ] デプロイ手順は再現可能か？

**進め方**:
1. 各観点でコードを精査し、発見事項を **3段階の重要度** で分類する:
   - **Critical（要修正）**: セキュリティリスク、データ損失の可能性、本番障害に直結
   - **Warning（改善推奨）**: 保守性の低下、パフォーマンス懸念、ベストプラクティスからの逸脱
   - **Info（参考）**: より良い方法の提案、モダンな書き方への移行提案
2. 各発見事項に以下を付記する:
   - **現状**: 今のコードがどうなっているか（該当ファイル・行を引用）
   - **問題**: なぜ問題か（具体的なリスクシナリオ）
   - **提案**: どう改善すべきか（コード例付き）
   - **学習ポイント**: この問題に関連する知識（なぜそれがベストプラクティスなのか）
3. 発見事項の一覧をユーザーに提示し、**対応方針を相談する**:
   - 「今回のドキュメントに改善後の設計を反映する」
   - 「現状のまま記録し、TODO として残す」
   - 「対応不要（意図的な設計判断）」

**出力例**:
```markdown
## コードレビュー結果

### Critical（要修正）
1. **[セキュリティ] APIキーのハードコード** (`src/config.py:23`)
   - 現状: `API_KEY = "sk-xxxx..."` が直接記述されている
   - 問題: リポジトリに push すると漏洩リスクがある
   - 提案: 環境変数 or Secret Manager に移行
   > **もっと学ぶなら:** 「12 Factor App 設定管理」で検索

### Warning（改善推奨）
2. **[アーキテクチャ] ルーティングにビジネスロジック混在** (`src/routes/user.py:45-89`)
   - 現状: ユーザー登録のバリデーション〜DB保存がルーティング関数内に全て記述
   - 問題: テストしにくい、再利用できない、変更時の影響範囲が大きい
   - 提案: Service層を分離（`src/services/user_service.py`）
   > **もっと学ぶなら:** 「レイヤードアーキテクチャ」「クリーンアーキテクチャ」
```

---

### Phase B2: 要件定義書の作成（逆引き＋改善提案付き）

**目的**: 既存コードから要件を逆引きし、「現状の実装がカバーしている要件」と「不足している要件」を明確にする。

**新規モードとの違い**:
- 各機能要件に **実装状況** カラムを追加する:

| 機能 | 優先度 | 実装状況 | 備考 |
|:---|:---|:---|:---|
| ユーザー登録 | 必須 | 実装済み | バリデーション不足（Phase B1 #3参照） |
| パスワードリセット | 必須 | 未実装 | セキュリティ上必要 |
| ログ出力 | 必須 | 部分的 | エラーログのみ、アクセスログなし |

- 各セクションに `> **改善提案:**` ブロックを追加する:
```markdown
> **改善提案:**
> 現在のコードではエラーハンドリングが個別の try-catch に依存しています。
> グローバルエラーハンドラを導入し、統一的なエラーレスポンス形式を定義することを推奨します。
> これにより、フロントエンド側でのエラー処理も簡素化できます。
```

**進め方**: 新規モードの Phase 2 と同じ構成で、上記の追加要素を含めて作成する。

---

### Phase B3: 技術選定の評価と改善提案

**目的**: 既存の技術スタックを評価し、現状維持すべきもの・移行を検討すべきものを整理する。

**進め方**:
1. Phase B0 で検出した技術スタックを一覧化する。
2. 各技術について **評価** を行う:

| 技術 | 現状 | 評価 | 改善案 | 理由 |
|:---|:---|:---|:---|:---|
| Express.js | v4.18 | 現状維持 | - | 安定版、十分な機能 |
| MongoDB | v5.0 | 要検討 | PostgreSQL移行 | リレーション多い設計にNoSQLは不適 |
| なし | テストなし | 要導入 | Jest + Supertest | 品質担保に必須 |

3. 各評価に教育コメントを付与する:
```markdown
> **なぜこの評価か？**
> 現在MongoDBを使っていますが、ユーザー・注文・商品間に明確な親子関係があり、
> JOIN相当の処理を $lookup で多用しています。この場合、RDB（PostgreSQL等）の方が
> クエリが直感的で、パフォーマンスも安定します。
> ただし、移行コストがあるため、段階的な移行計画が必要です。

> **もっと学ぶなら:**
> - 「MongoDB vs PostgreSQL 使い分け」で検索
> - Martin Fowler の「NoSQL Distilled」が判断基準として優秀
```

---

### Phase B4: 仕様書の作成（現状＋改善後の姿）

**目的**: 現在の実装を正確に文書化しつつ、改善後の理想的な仕様も併記する。

**新規モードとの違い**:
- システム構成図を **2つ** 作成する:
  - `### 現状のシステム構成図` — 今のコードが実際にどう動いているか
  - `### 改善後のシステム構成図（提案）` — Phase B1 の改善提案を反映した理想形
- シーケンス図も同様に、現状と改善後を対比で示す（改善箇所にコメント付与）
- 各機能仕様に **現状の実装** と **推奨する改善** を併記する:

```markdown
#### 3.1.3. 処理仕様

**現状の実装:**
1. ルーティング関数内でバリデーション
2. 同じ関数内でDB保存
3. レスポンス返却

> **改善提案:**
> 1. バリデーション → Middleware / Schema層に分離
> 2. ビジネスロジック → Service層に分離
> 3. DB操作 → Repository層に分離
> 4. レスポンス返却
>
> **なぜこうすべきか？**
> 関心の分離（Separation of Concerns）により、各層を独立してテスト可能になります。
> バリデーションのルール変更がDB操作のコードに影響しなくなり、保守性が大幅に向上します。
>
> **もっと学ぶなら:**
> - 「MVC パターン」「レイヤードアーキテクチャ」で検索
> - 「Clean Architecture（Robert C. Martin）」の概念図が分かりやすい
```

---

### Phase B5: DB設計書の作成（現状分析＋改善提案）

**目的**: 既存のDB設計を文書化し、正規化・インデックス・制約の観点から改善点を提示する。

**進め方**:
1. **既存スキーマの読み取り**: マイグレーションファイル、ORMモデル定義、または直接スキーマから現状のテーブル構造を抽出する。
2. **現状のER図** を Mermaid で作成する。
3. **テーブル定義** を表形式で記述する（新規モードと同じ形式）。
4. **DB設計レビュー** を以下の観点で実施し、発見事項を記載する:

| 観点 | チェック内容 |
|:---|:---|
| 正規化 | 第3正規形を満たしているか？非正規化が意図的か？ |
| 命名 | テーブル名・カラム名の規則は統一されているか？ |
| 型選定 | データの性質に対して適切な型か？（例: 金額に FLOAT を使っていないか） |
| 制約 | NOT NULL / UNIQUE / FK が適切に設定されているか？ |
| インデックス | よく検索されるカラムにインデックスがあるか？不要なインデックスはないか？ |
| 共通カラム | `created_at` / `updated_at` は全テーブルにあるか？ |
| ソフトデリート | 論理削除 vs 物理削除の方針は明確か？ |

5. 改善提案を含む **改善後のER図（提案）** を併記する。

**Mode C への遷移**:
Phase B5 完了後、以下を提示する:
```
設計書（要件定義書・仕様書・DB設計書・コードレビューレポート）が完成しました。
このまま実装（改善含む）に進みますか？

→ はい: 実装モード（Mode C）の Phase C0 に進みます
→ いいえ: ここで終了します（後から Mode C を呼び出せます）
```

**出力例**:
```markdown
### 3.1. 現状のER図
（現在のコードから読み取ったER図）

### 3.2. DB設計レビュー結果

| # | 対象 | 重要度 | 指摘内容 | 改善案 |
|:---|:---|:---|:---|:---|
| 1 | `users.phone` | Warning | VARCHAR(255) は過剰。電話番号は最大15桁 | VARCHAR(20) に変更 |
| 2 | `orders` テーブル | Warning | `updated_at` カラムなし | 追加推奨（更新追跡に必要） |
| 3 | `orders.total_price` | Critical | FLOAT型 → 浮動小数点の丸め誤差リスク | DECIMAL(10,2) に変更 |
| 4 | `products.category` | Info | 文字列直書き → マスタテーブル化推奨 | `categories` テーブルを新設しFK参照 |

> **なぜ金額に FLOAT を使ってはいけないか？**
> FLOAT/DOUBLE は内部的に2進数で小数を表現するため、10進数の小数を正確に表現できません。
> 例: 0.1 + 0.2 = 0.30000000000000004 になることがあります。
> 金額計算では1円の誤差も許されないため、DECIMAL型（固定小数点）を使います。
>
> **もっと学ぶなら:**
> - 「浮動小数点 誤差 なぜ」で検索すると原理が分かります
> - IEEE 754 の基礎を知っておくと、他の場面でも役立ちます

### 3.3. 改善後のER図（提案）
（改善を反映したER図）
```

# 実装モード (Mode C)

作成済みの設計書（要件定義書・仕様書・DB設計書）を元に、**タスク分解→TDD実装→検証** の一貫フローで開発を遂行する。
ユーザーが開発過程を把握しながら1機能ずつ実装を進め、機能追加も容易に行える下流特化のワークフロー。

本モードは **5つのフェーズ** を順番に進める。各フェーズの最後でユーザーの承認を得てから次に進む。

---

### Phase C0: 設計ドキュメント読み込み & 実装計画

**目的**: `docs/` 内の3つの設計書を解析し、実装可能なタスク単位に分解する。

**前提条件**:
- `docs/` に `*_requirements_*.md`, `*_specification_*.md`, `*_database_design_*.md` が存在すること。
- 存在しない場合はエラーメッセージを表示し、Mode A または Mode B の実行を案内する。

**進め方**:
1. `docs/` から設計書ファイルを自動検出し、最新バージョンを読み込む。
2. 以下の情報を設計書から抽出する:
   - 要件定義書: 機能一覧と優先度
   - 仕様書: 技術スタック、システム構成、機能仕様の詳細、処理フロー
   - DB設計書: テーブル定義、ER図、マイグレーション方針
3. 抽出した情報を基に、**実装順序** を決定する:
   - **Layer 1（土台）**: プロジェクトセットアップ、DBスキーマ & マイグレーション
   - **Layer 2（基盤）**: 認証・認可（存在する場合）
   - **Layer 3（コア）**: 必須機能（優先度「必須」のもの、依存関係順）
   - **Layer 4（補助）**: 優先度「あれば良い」の機能
   - **Layer 5（品質）**: 非機能要件対応（パフォーマンス最適化、ロギング、監視等）
4. 各タスクを以下のフォーマットで分解する:

```markdown
## 実装タスク一覧

| # | タスク名 | 対応セクション | 優先度 | 依存 | 状態 | 備考 |
|:--|:---------|:---------------|:-------|:-----|:-----|:-----|
| 1 | プロジェクトセットアップ | 仕様書 2.3 | 必須 | - | 未着手 | スキャフォールド+依存 |
| 2 | DBスキーマ作成 | DB設計書 全体 | 必須 | #1 | 未着手 | マイグレーション含む |
| 3 | ユーザー認証API | 仕様書 3.1 | 必須 | #2 | 未着手 | JWT/Session |
| 4 | 商品一覧API | 仕様書 3.2 | 必須 | #2 | 未着手 | ページネーション含む |
| ... | | | | | | |
```

5. **実装計画書** (`<slug>_implementation_plan_v1.md`) を `docs/` に出力する（`references/implementation-plan-template.md` に準拠）。
6. ユーザーに提示し、タスクの順序・粒度・スコープの調整を受ける。

**教育ポイント**:
```markdown
> **なぜこの順番で実装するか？**
> ソフトウェアには「依存関係」があります。DBスキーマがなければAPIは書けず、
> 認証がなければ保護されたエンドポイントは作れません。
> 依存の下流（土台）から順に積み上げることで、各ステップで動作確認が可能になります。

> **もっと学ぶなら:**
> - 「トポロジカルソート」で検索すると、依存関係の整理方法が学べます
> - 「Vertical Slice Architecture」も、機能単位で実装を進める考え方として参考になります
```

---

### Phase C1: プロジェクトセットアップ

**目的**: 仕様書の技術選定に基づき、プロジェクトの骨格を構築する。

**進め方**:
1. 仕様書の「システム構成」「技術選定理由」セクションから技術スタックを読み取る。
2. 以下のステップを順番に実行する（各ステップ完了時にユーザーに報告）:

| ステップ | 内容 | 出力物 |
|:---------|:-----|:-------|
| 1. ディレクトリ構成 | 仕様書のレイヤー設計・命名規則に準拠した構成を作成 | プロジェクトフォルダ一式 |
| 2. パッケージ初期化 | `package.json` / `pyproject.toml` 等の作成、依存パッケージのインストール | 依存定義ファイル |
| 3. DB セットアップ | マイグレーションツール導入 + DB設計書に基づく初期スキーマ作成 | マイグレーションファイル |
| 4. テスト基盤 | テストフレームワーク導入 + カバレッジ閾値設定（80%） | テスト設定ファイル |
| 5. Lint / Format | コード品質ツールの導入と設定 | `.eslintrc` / `ruff.toml` 等 |
| 6. 環境変数 | `.env.example` の作成（仕様書の Secret Manager 項目に準拠） | `.env.example` |
| 7. Git 初期設定 | `.gitignore` 作成、仕様書のブランチ戦略に準拠 | `.gitignore` |

3. 全ステップ完了後、**ビルドとテスト（空のテストスイート）が通る状態** であることを確認する。
4. ユーザーに完了報告し、承認を得る。

**教育ポイント**:
```markdown
> **なぜ最初にテスト基盤をセットアップするか？**
> 後からテストを導入すると、「テストしにくいコード」が大量に生まれた後に苦労します。
> 最初からテスト基盤があれば、TDD（テスト駆動開発）で進められるため、
> 品質を保ちながら開発速度も維持できます。

> **もっと学ぶなら:**
> - 「テスト駆動開発」Kent Beck 著 — TDDのバイブル
> - 「テストピラミッド」で検索すると、テストの種類と比率が学べます
```

---

### Phase C2: 機能実装（イテレーティブ TDD サイクル）

**目的**: 実装計画のタスクを1つずつ、TDD（テスト駆動開発）で実装する。

**1タスクあたりのサイクル**:

```
┌─────────────────────────────────────────────────┐
│  Step 1: タスクスコープ確認                        │
│    → 仕様書の対応セクションを引用表示               │
│    → 「タスク #N: 〇〇 を実装します」と宣言         │
│    → ユーザーに追加の要望・変更がないか確認          │
├─────────────────────────────────────────────────┤
│  Step 2: テスト作成（RED フェーズ）                │
│    → 仕様書の入出力仕様からテストケースを生成        │
│    → 正常系 + エッジケース + エラーケース           │
│    → テスト実行 → 全て FAIL であることを確認        │
│    → 作成したテストケースをユーザーに報告            │
├─────────────────────────────────────────────────┤
│  Step 3: 実装（GREEN フェーズ）                    │
│    → テストを通す最小限のコードを書く               │
│    → テスト実行 → 全て PASS であることを確認        │
├─────────────────────────────────────────────────┤
│  Step 4: リファクタ（REFACTOR フェーズ）            │
│    → コード品質を改善（テストは GREEN を維持）       │
│    → 重複排除、命名改善、構造の整理                 │
├─────────────────────────────────────────────────┤
│  Step 5: ミニ検証                                │
│    → Build チェック                               │
│    → Type チェック（型付き言語の場合）               │
│    → Lint チェック                                │
│    → テストスイート全体を実行（リグレッション確認）    │
│    → 問題があれば即座に修正                        │
├─────────────────────────────────────────────────┤
│  Step 6: タスク完了報告 & 承認                     │
│    → 実装内容のサマリーをユーザーに報告              │
│    → 作成/変更したファイル一覧                      │
│    → 実装計画書のステータスを「完了」に更新           │
│    → ユーザー承認後、次のタスクへ                   │
└─────────────────────────────────────────────────┘
```

**テストケース設計のガイドライン**:
- **正常系**: 仕様書の処理仕様に基づく期待動作
- **境界値**: 最小値、最大値、空値、長大な入力
- **異常系**: 仕様書のエラーケースに基づく異常パターン
- **テストの独立性**: 各テストは他のテストに依存しない（setUp/tearDownで状態リセット）

**テストの種類**（プロジェクトの技術スタックに応じて選択）:
| テスト種類 | 対象 | 目安 |
|:-----------|:-----|:-----|
| ユニットテスト | 個別の関数・メソッド・ユーティリティ | 全タスクで必須 |
| インテグレーションテスト | APIエンドポイント、DB操作、サービス間連携 | APIタスクで必須 |
| E2Eテスト | ユーザーフロー全体 | コアフロー完成時に追加 |

**実装計画書のリアルタイム更新**:
各タスク完了時に `<slug>_implementation_plan_v*.md` の該当タスクの状態カラムを更新する。
進捗サマリーをユーザーに報告する:

```markdown
## 進捗: 3/12 タスク完了 (25%)
[███░░░░░░░░░]

| 完了 | 進行中 | 未着手 |
|:-----|:-------|:-------|
| #1 プロジェクトセットアップ | #4 商品一覧API | #5 注文API |
| #2 DBスキーマ作成 | | #6 決済連携 |
| #3 ユーザー認証API | | ... |
```

**教育ポイント**（初回のタスクでのみ詳細に解説する）:
```markdown
> **TDD の RED → GREEN → REFACTOR サイクルとは？**
> 1. RED: まずテストを書く（実装がないのでテストは失敗する = 赤）
> 2. GREEN: テストを通す最小限のコードを書く（テストが通る = 緑）
> 3. REFACTOR: コードを綺麗にする（テストが通り続けることを確認しながら）
>
> この順番を守ることで:
> - テスト可能な設計が自然に生まれる
> - 過剰な実装（YAGNI違反）を防げる
> - リファクタリングの安全ネットが常にある

> **もっと学ぶなら:**
> - 「テスト駆動開発」Kent Beck 著
> - 「RED GREEN REFACTOR」で検索するとサイクルの図解が見つかります
```

---

### Phase C3: 統合検証 & 品質ゲート

**目的**: 全機能の実装完了後（または任意のマイルストーントで）、6段階の品質チェックを実施する。

**検証フェーズ**:

| # | 検証項目 | 内容 | 合格基準 |
|:--|:---------|:-----|:---------|
| 1 | Build | プロジェクトがエラーなくビルドできるか | エラー 0 |
| 2 | Type Check | 型チェック通過（TypeScript: `tsc --noEmit`, Python: `pyright` 等） | エラー 0 |
| 3 | Lint | リンター準拠（ESLint / Ruff 等） | エラー 0（Warning は許容） |
| 4 | Test Suite | 全テスト通過 + カバレッジ閾値 | 全 PASS + 80% 以上 |
| 5 | Security Scan | ハードコード秘密情報、console.log/print デバッグ文の残存 | 検出 0 |
| 6 | 設計整合性 | 仕様書の全機能が実装済みか、DB設計書とスキーマが一致しているか | 全機能カバー |

**実行手順**:
1. 各検証項目を順番に実行する。
2. **Phase 1（Build）が失敗した場合は即座に停止** し、修正を優先する。
3. 他の項目で問題が見つかった場合は、一覧化して報告する。
4. 全項目 PASS 後、検証レポートを出力する。

**出力**: 検証レポート (`<slug>_verification_report_v1.md`) を `docs/` に出力する（`references/verification-report-template.md` に準拠）。

```markdown
## 検証レポート
===================

Build:     [PASS]
Types:     [PASS] (0 errors)
Lint:      [PASS] (2 warnings)
Tests:     [PASS] (47/47 passed, 84% coverage)
Security:  [PASS] (0 issues)
設計整合:   [PASS] (12/12 機能実装済み)

総合判定:   READY

残課題:
- Lint warning 2件（非クリティカル、対応推奨）
```

**問題がある場合の対応フロー**:
1. 問題一覧をユーザーに提示する。
2. 各問題の修正方針を提案する。
3. ユーザー承認後、修正を実施する。
4. 再度検証を実行する（修正した項目のみ or 全項目）。

---

### Phase C4: 機能追加ワークフロー

**目的**: 初期実装完了後に新しい機能を追加する際の、**設計→実装の一貫フロー**。
設計書を先に更新してから実装に入ることで、ドキュメントとコードの一貫性を維持する。

**進め方**:

```
┌──────────────────────────────────────────────────────┐
│  Step 1: 追加要件ヒアリング（軽量版）                    │
│    → Phase 1 の簡易版（1〜2往復で完了）                 │
│    → 追加機能の目的、対象ユーザー、優先度を確認          │
│    → 既存機能への影響範囲を特定                         │
├──────────────────────────────────────────────────────┤
│  Step 2: 設計ドキュメント更新                            │
│    → 要件定義書: 機能要件セクションに新機能を追記         │
│    → 仕様書: 機能仕様セクションを追加、シーケンス図を更新  │
│    → DB設計書: テーブル/カラム変更がある場合はER図も更新   │
│    → 各ドキュメントのバージョンをインクリメント（v1→v2）  │
│    → ユーザーに変更内容を提示し承認を得る                │
├──────────────────────────────────────────────────────┤
│  Step 3: 実装計画更新                                   │
│    → 新タスクを実装計画書に追加                          │
│    → 既存タスクとの依存関係を整理                        │
│    → 実装計画書のバージョンをインクリメント               │
├──────────────────────────────────────────────────────┤
│  Step 4: TDD 実装サイクル                               │
│    → Phase C2 と同じ RED→GREEN→REFACTOR サイクルで実装  │
│    → 各タスク完了時にミニ検証を実施                      │
├──────────────────────────────────────────────────────┤
│  Step 5: 統合検証                                       │
│    → Phase C3 と同じ6段階の品質ゲートを実行              │
│    → 特に重要: 既存機能のリグレッションがないことを確認    │
│    → 検証レポートのバージョンをインクリメント             │
└──────────────────────────────────────────────────────┘
```

**設計書更新時のルール**:
- **既存セクションの変更**: 変更箇所に `<!-- v2 で追加 -->` 等のコメントを付与し、変更履歴を追跡可能にする。
- **新規セクションの追加**: 既存の番号体系に従って追番する。
- **DB変更**: 新しいマイグレーションファイルとして追加する（既存マイグレーションは編集しない）。

**教育ポイント**:
```markdown
> **なぜ「コードだけ追加」ではなく設計書を先に更新するか？**
> 設計書を更新せずにコードだけ変更すると、時間が経つにつれて
> 「ドキュメントと実際のコードが全く違う」状態になります（ドキュメント腐敗）。
> 設計書を先に更新する習慣をつけることで:
> - 変更の影響範囲を事前に把握できる
> - チームメンバーが最新の仕様をいつでも参照できる
> - 後から参加したメンバーのオンボーディングが容易になる

> **もっと学ぶなら:**
> - 「Documentation as Code」で検索
> - 「ADR（Architecture Decision Records）」も設計判断の記録方法として有用です
```

## Output Rules

### ファイル出力
- 出力先: スキルを呼び出したリポジトリの `docs/` フォルダ
- 新規モードのファイル名: `<slug>_requirements_v1.md`, `<slug>_specification_v1.md`, `<slug>_database_design_v1.md`
- 分析モードのファイル名: `<slug>_requirements_v1.md`, `<slug>_specification_v1.md`, `<slug>_database_design_v1.md`, `<slug>_code_review_v1.md`
- 分析モードでは追加で **コードレビューレポート** (`<slug>_code_review_v1.md`) を出力する
  - Phase B1 の全発見事項を重要度別に整理
  - 対応方針（修正する / TODO / 対応不要）をユーザーとの合意に基づいて記録
- 実装モードのファイル名: `<slug>_implementation_plan_v1.md`, `<slug>_verification_report_v1.md`
- 実装モードでは以下を出力する:
  - **実装計画書** (`<slug>_implementation_plan_v1.md`): タスク一覧と進捗管理（`references/implementation-plan-template.md` に準拠）
  - **検証レポート** (`<slug>_verification_report_v1.md`): 品質ゲートの結果（`references/verification-report-template.md` に準拠）
  - 実装計画書はタスク完了ごとにリアルタイムで状態を更新する
- 既存ファイルがある場合はバージョン番号をインクリメント（v2, v3...）
- 機能追加時（Phase C4）は既存の設計書・実装計画書・検証レポートのバージョンをインクリメントして更新する

### 教育コメントの書式
ドキュメント内の教育コメントは以下の書式で統一する:

```markdown
> **なぜこうしたか？**
> 〇〇と△△を比較検討しました。今回は□□という要件があるため、〇〇を採用しています。
> △△は◆◆のようなケースで強みを発揮するため、将来的に検討の余地があります。

> **もっと学ぶなら:**
> - 「〇〇」で検索すると基礎が学べます
> - 書籍『△△』の第N章が参考になります
> - 公式ドキュメント: [リンク先の説明]
```

### 対話のルール
- 各フェーズの開始時に、「これからPhase N: 〇〇を始めます」と宣言する。
- 専門用語を使うときは、初出時に括弧書きで簡単な説明を添える。
- 質問は一度に3〜5個に留め、ユーザーを圧倒しない。
- 「わからない」「後で決めたい」という回答には仮置きで対応し、先に進む。
- 各フェーズの最後に「このフェーズの内容で問題なければ、次に進みます」と確認する。

### Mermaid図の利用
- システム構成図、シーケンス図、ER図は **Mermaid記法** で記述する。
- 図の中の表記は日本語を基本とする（技術用語は英語可）。

## Constraints

- 対象規模: 個人開発〜7人チーム
- 言語: 日本語で出力
- 教育レベル: 「なぜRDBを選んだか」程度の粒度で技術選定理由を説明
- テンプレート詳細は `references/` フォルダのファイルを参照
- ドキュメント構成は参照プロジェクト (`Sakura Rin/docs`) の品質水準に準拠
