# 実装計画書テンプレート

本テンプレートは `project-design-guide` スキルの実装モード（Mode C）で
実装計画書を生成する際の構成ガイドラインである。

---

## ドキュメント構成

### 1. 概要

```markdown
## 1. 概要

| 項目 | 内容 |
|:---|:---|
| プロジェクト名 | `<project-name>` |
| 参照: 要件定義書 | `docs/<slug>_requirements_v1.md` |
| 参照: 仕様書 | `docs/<slug>_specification_v1.md` |
| 参照: DB設計書 | `docs/<slug>_database_design_v1.md` |
| 技術スタック | <仕様書から抽出> |
| 作成日 | YYYY-MM-DD |
| 最終更新日 | YYYY-MM-DD |
```

### 2. 実装方針

以下を記述する:
- **開発アプローチ**: TDD（テスト駆動開発）で進める旨を明記
- **テスト戦略**: ユニットテスト + インテグレーションテスト（+ E2E テスト）
- **カバレッジ目標**: 80% 以上
- **ブランチ戦略**: 仕様書の記載に準拠
- **品質ゲート**: Phase C3 の6段階検証を適用

### 3. 依存関係グラフ

Mermaid `graph TD` でタスク間の依存関係を視覚化する:

```markdown
### 3. 依存関係グラフ

（Mermaid graph TD でタスク依存を図示）

例:
graph TD
    T1[#1 プロジェクトセットアップ] --> T2[#2 DBスキーマ作成]
    T2 --> T3[#3 ユーザー認証API]
    T2 --> T4[#4 商品一覧API]
    T3 --> T5[#5 注文API]
    T4 --> T5
    T5 --> T6[#6 決済連携]
```

### 4. 実装タスク一覧

タスクを表形式で管理する:

```markdown
## 4. 実装タスク一覧

| # | タスク名 | 対応セクション | レイヤー | 優先度 | 依存 | 状態 | 備考 |
|:--|:---------|:---------------|:---------|:-------|:-----|:-----|:-----|
| 1 | プロジェクトセットアップ | 仕様書 2.3 | Layer 1 | 必須 | - | 未着手 | スキャフォールド |
| 2 | DBスキーマ作成 | DB設計書 全体 | Layer 1 | 必須 | #1 | 未着手 | マイグレーション含む |
| 3 | ユーザー認証API | 仕様書 3.1 | Layer 2 | 必須 | #2 | 未着手 | JWT認証 |
| ... | | | | | | | |
```

**レイヤーの定義**:

| レイヤー | 内容 | 実装順 |
|:---------|:-----|:-------|
| Layer 1（土台） | プロジェクトセットアップ、DBスキーマ & マイグレーション | 最初 |
| Layer 2（基盤） | 認証・認可 | 土台の次 |
| Layer 3（コア） | 必須機能（優先度「必須」） | 基盤の次 |
| Layer 4（補助） | 優先度「あれば良い」の機能 | コアの次 |
| Layer 5（品質） | 非機能要件（パフォーマンス、ロギング、監視） | 最後 |

**状態の定義**:

| 状態 | 意味 |
|:-----|:-----|
| 未着手 | まだ着手していない |
| 進行中 | TDD サイクル実行中 |
| 完了 | 実装 + ミニ検証 PASS |
| ブロック | 依存タスクが未完了 or 外部要因で停止 |

### 5. 進捗サマリー

リアルタイムで更新するセクション:

```markdown
## 5. 進捗サマリー

**全体進捗**: 0/N タスク完了 (0%)

|:-----|:-------|
```

## タスク分解のガイドライン

### 粒度の基準
- **1タスク = 1つの明確な成果物** が出るサイズ
- 目安: 1タスクで作成/変更するファイルが 3〜8 ファイル程度
- 大きすぎるタスクは分割する（例: 「商品管理」→「商品一覧API」「商品詳細API」「商品登録API」）

### 依存関係の整理ルール
- DBスキーマは全APIタスクの前提
- 認証機能は認証が必要なAPIの前提
- フロントエンドのタスクは対応するAPIタスクの後
- E2Eテストは関連する機能の全タスク完了後

### 仕様書との紐付け
- 各タスクの「対応セクション」には仕様書のセクション番号を記載する
- これにより、タスクの完了 = 仕様書の該当機能が実装済み、を追跡できる
